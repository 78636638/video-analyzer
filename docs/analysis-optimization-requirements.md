# 视频分析系统视觉与音频协同优化需求文档

## 1. 文档目的

本文档基于当前项目的真实代码结构与现有能力，提出一套针对“图片视觉识别慢、长视频上下文膨胀、输出连贯性与人物一致性不足”的优化需求方案，并补齐音频处理与图片识别之间的联动规则。

文档目标：

- 在不脱离现有项目架构的前提下，提出可落地的优化需求。
- 明确当前系统真实行为、现有瓶颈与改造边界。
- 建立图片分析、音频转写、剧情连贯性、人物一致性之间的统一规则。
- 确保方案具备统一入口、统一收口、可回退、可兼容、可评审的闭环设计。

说明：

- 本文档是需求文档，不直接替代代码实现方案。
- 涉及核心链路、提示词结构、输出结构、上下文记忆策略的改动，均需多人评审后再实施。

## 2. 当前系统真实架构

根据当前代码，系统主链路如下：

1. `video_analyzer/cli.py`
   - 系统主入口。
   - 负责配置加载、音频处理、关键帧抽取、逐帧识别、最终重建、结果输出。

2. `video_analyzer/frame.py`
   - 负责视频关键帧抽取。
   - 当前基于帧差分进行关键帧候选筛选，并按得分选取关键帧。

3. `video_analyzer/audio_processor.py`
   - 负责音频提取与转写。
   - 使用 `ffmpeg` 提取音频，失败时降级使用 `pydub`。
   - 使用 `faster-whisper` 转写，输出整段文本与分段时间戳。

4. `video_analyzer/analyzer.py`
   - 负责逐帧视觉分析与最终视频重建。
   - 逐帧分析时会把历史帧描述持续拼接到下一次 prompt。
   - 最终重建时会把全部帧分析结果与完整转写文本拼入总结 prompt。

5. `video_analyzer/clients/ollama.py`
   - 负责向 Ollama `/api/generate` 发起视觉模型推理请求。
   - 当前每次只发送当前图片，但会携带完整 prompt 文本。

6. `video_analyzer/prompts/frame_analysis/frame_analysis.txt`
   - 逐帧分析提示词模板。
   - 含 `{PREVIOUS_FRAMES}` 与 `{prompt}` 占位符。

7. `video_analyzer/prompts/frame_analysis/describe.txt`
   - 整片视频总结提示词模板。
   - 含 `{FRAME_NOTES}`、`{TRANSCRIPT}`、`{prompt}` 占位符。

## 3. 当前系统真实行为说明

### 3.1 图片识别当前行为

当前系统不是把“之前的图片”重复送入模型，而是：

- 每次只发送“当前这一张图片”。
- 但会把“之前所有帧的文字分析结果”拼接到下一次 prompt 中。

对应行为位于 `video_analyzer/analyzer.py`：

- `self.previous_analyses` 保存全部历史帧分析。
- `analyze_frame()` 每次调用时，使用 `_format_previous_analyses()` 把历史描述全部展开。
- 随着帧数增加，prompt 长度线性增长。

### 3.2 音频处理当前行为

当前系统音频链路如下：

1. `AudioProcessor.extract_audio()`
   - 从视频中提取 `audio.wav`。
   - 若视频无音轨，则返回 `None`。

2. `AudioProcessor.transcribe()`
   - 使用 `faster-whisper` 转写。
   - 输出：
     - `text`
     - `segments`
     - `language`

3. `VideoAnalyzer.reconstruct_video()`
   - 仅在最终重建阶段将 `transcript.text` 注入 `{TRANSCRIPT}`。

当前音频没有参与：

- 逐帧分析阶段的上下文提示。
- 人物说话归属判断。
- 场景切换辅助判断。
- 关键剧情节点提炼。

### 3.3 当前性能问题本质

当前长视频慢的核心原因不是图片编码本身，而是以下因素叠加：

1. 逐帧 prompt 持续增长。
2. 视觉模型使用 `llava:13b`，推理本身较重。
3. 每帧都以完整大 prompt 重新推理，没有上下文压缩机制。
4. 音频只在最终阶段使用，无法前置参与剧情约束，导致视觉 prompt 承担过多叙事记忆任务。
5. 当前抽帧密度偏高时，会放大后续全部成本。

## 4. 现存问题清单

### 4.1 性能问题

- 单帧识别耗时会随视频长度增加而逐步上升。
- 长视频容易触发模型上下文截断，导致 prompt 被裁剪。
- 关键帧数量一多，整段任务时长失控。

### 4.2 连贯性问题

- 历史描述全量叠加虽然理论上能保持连贯，但一旦被截断，反而导致剧情记忆不稳定。
- 单纯依赖自由文本历史，不利于人物、场景、道具的一致性维护。

### 4.3 音视频割裂问题

- 音频仅在最终总结阶段使用，没有与帧级分析建立规则联动。
- 无法借助台词、旁白、环境音对当前画面分析进行约束。

### 4.4 维护问题

- 当前上下文管理逻辑全部隐含在 prompt 拼接中，没有统一的结构化状态对象。
- 后续如果继续叠加优化，很容易出现多路调用、逻辑分叉、难以维护的问题。

## 5. 优化目标

### 5.1 核心目标

- 显著降低长视频逐帧视觉识别总耗时。
- 保持甚至增强输出内容的剧情连贯性。
- 提升人物一致性、场景一致性、道具一致性。
- 建立音频与图片识别的协同规则。
- 让分析结果不仅可读，还能沉淀为可供后续剧本生成、分镜整理、视频再创作使用的结构化素材。

### 5.2 架构目标

- 保持 `cli.py` 为统一主入口。
- 保持 `VideoAnalyzer` 为统一分析收口点。
- 所有上下文状态通过统一对象管理，不允许多套并行上下文机制。
- 配置统一进入 `default_config.json/config.json`，禁止散落硬编码。

### 5.3 兼容目标

- 保留现有命令行使用方式。
- 保留现有输出主结构字段，新增字段尽量采用可选扩展方式。
- UI 与 CLI 读取配置的行为保持一致。

## 6. 总体优化方案

系统优化采用“统一上下文状态 + 分层记忆 + 音视频协同 + 分阶段重建”的方案。

### 6.1 统一状态中心

新增统一分析状态对象，逻辑上由 `VideoAnalyzer` 统一持有与维护，建议命名方向如下：

- `analysis_state`
- `story_memory`
- `character_registry`
- `recent_frame_window`
- `audio_memory`

约束：

- 不允许在 `cli.py`、prompt 文件、client 层分别维护各自独立状态。
- 所有需要参与 prompt 构造的上下文，必须从统一状态对象读取。

### 6.2 分层记忆

将当前“全历史逐字拼接”改为三层记忆：

1. 当前帧信息
   - 当前图片。
   - 当前时间戳。
   - 当前场景判断。

2. 局部窗口记忆
   - 最近 `K` 帧的结构化摘要。
   - 服务于动作连续性、镜头衔接。

3. 全局剧情记忆
   - 已出现人物。
   - 场景主线。
   - 关键剧情节点。
   - 重要道具与关系。

目标：

- 保证 prompt 长度有上限。
- 保留必要的剧情连续性。
- 避免越到后面越慢。

### 6.3 分块分析与分块总结

为长视频引入分块处理思想：

- 按连续时间段或场景切换划分为若干分析块。
- 块内采用局部窗口分析。
- 块结束后生成块级摘要，写入全局剧情记忆。
- 最终重建不再仅依赖全量逐帧原文，而是依赖：
  - 帧级结果
  - 块级摘要
  - 全局剧情记忆
  - 音频摘要

### 6.4 音视频协同

音频与图片不再仅在最终总结阶段汇合，而是要在中间分析阶段建立规则化联动：

- 音频先转写。
- 转写结果拆成可检索的时间段片段。
- 每帧分析时按时间窗口查找附近音频片段，作为辅助上下文。
- 音频只提供约束与补充，不直接覆盖视觉事实。

## 7. 视觉识别优化需求

### 7.1 有界上下文窗口

需求：

- 逐帧分析时，不再将全部历史帧描述完整拼接进 prompt。
- 改为仅注入最近 `K` 帧的结构化摘要。

建议默认策略：

- `context_window = 5` 或 `8`

需求说明：

- `K` 必须可配置。
- `K=0` 时允许退化为“只看当前帧 + 全局摘要”。
- 必须保留回退兼容开关，以便与旧策略对比。

### 7.2 结构化帧分析输出

需求：

- 单帧分析结果不能只保留自由文本。
- 需同时产出结构化字段，至少包括：
  - `scene`
  - `characters`
  - `actions`
  - `objects`
  - `dialogue_hint`
  - `continuity_points`
  - `raw_response`

目的：

- 后续摘要可基于结构化字段压缩。
- 人物一致性与剧情连续性有稳定抓手。

### 7.3 场景切换识别

需求：

- 复用现有帧差分能力，增加“场景切换”判断规则。
- 当检测到明显场景切换时：
  - 局部窗口可重置或缩短。
  - 块级摘要应立即更新。

目标：

- 防止上一场景信息污染下一场景。

### 7.4 图片输入优化

需求：

- 在统一图片编码入口增加可配置的缩放策略。
- 发送给模型前按最长边进行缩放。

目标：

- 降低图片编码与传输成本。
- 在不明显损失有效信息的前提下提高吞吐。

约束：

- 必须在统一入口处理，不能不同 client 各自实现不同缩放逻辑。

### 7.5 自适应抽帧

需求：

- 在现有关键帧抽取基础上，引入自适应密度控制。
- 静态/低变化段降低抽帧频率。
- 高变化/动作段保留更高密度。

目标：

- 从源头减少无效视觉推理次数。

## 8. 音频处理优化需求

### 8.1 音频转写继续保留现有主链路

继续保留：

- `ffmpeg` 优先提取音频。
- `pydub` 作为回退。
- `faster-whisper` 负责转写。

原因：

- 该链路已存在且与当前项目兼容，不建议重构为其他音频技术栈。

### 8.2 音频摘要层

新增需求：

- 在完整转写基础上，增加音频摘要层。

音频摘要需要输出：

- 时间段摘要
- 关键对白
- 角色说话提示
- 明显环境音
- 情绪/氛围关键词

目标：

- 避免把整段 transcript 原文长期直接塞给后续 prompt。
- 为视觉分析与最终总结提供更短、更稳定的语义上下文。

### 8.3 音频片段对齐

新增需求：

- 每个关键帧分析时，按时间戳查询附近音频片段。
- 关联范围建议支持配置，如：
  - `frame.timestamp +/- 2s`
  - 或按分段边界自动吸附

目的：

- 为当前帧提供同时刻对白/环境音辅助。
- 增强人物正在说话、场景正在发生什么的判断。

### 8.4 音频质量降级策略

当音频缺失、音频质量差、无可靠转写时，必须支持以下降级：

- 仍可继续纯视觉分析。
- 在最终结果中明确标记音频不可用或音频可信度不足。
- 不能因音频失败中断主流程。

## 9. 音频与图片识别联动规则

本节为本次需求文档必须新增的统一规则。

### 9.1 联动原则

- 视觉是事实主源。
- 音频是补充主源。
- 当视觉与音频存在冲突时：
  - 优先保留视觉直接可见事实。
  - 音频内容作为候选解释，不得强行覆盖视觉结论。

### 9.2 帧级联动规则

每个关键帧分析时，应同时注入：

- 当前图片
- 最近 `K` 帧结构化上下文
- 当前全局剧情记忆
- 当前时间窗口对应的音频片段摘要

用于支持：

- 当前角色可能正在说话的判断
- 镜头外事件的辅助理解
- 人物关系与对白内容的一致性补充

### 9.3 人物一致性规则

若音频中出现稳定称谓、名字、关系词，应写入人物注册表，作为后续视觉分析的辅助约束：

- 示例类型：
  - 人名
  - 称呼
  - 关系词
  - 职务/身份

但约束规则必须是：

- 仅当后续视觉描述不冲突时才增强使用。
- 不能因为音频中提到某人，就强行认定画面中的人物一定是该人。

### 9.4 场景连续性规则

若画面变化小、但音频对白连续：

- 优先视为同一场景延续。

若画面变化大、且音频也发生明显切换：

- 可视为新场景或新分析块。

### 9.5 最终总结规则

最终视频总结必须综合以下信息源：

- 帧级结构化结果
- 块级剧情摘要
- 全局剧情记忆
- 音频摘要
- 原始 transcript

规则：

- transcript 原文不再作为唯一音频输入。
- transcript 原文主要用于校验与补充，不应承担全部叙事主干。

## 10. 统一入口、统一收口设计

### 10.1 统一入口

维持 `video_analyzer/cli.py` 为唯一业务主入口。

要求：

- 音频抽取、音频转写、关键帧抽取、逐帧识别、块级摘要、最终总结，均由同一条 CLI 主链路编排。
- UI 层只负责参数提交与结果展示，不承载分析逻辑。

### 10.2 统一收口

维持 `VideoAnalyzer` 为统一分析收口点。

要求：

- 帧级上下文、人物注册表、剧情记忆、音频记忆统一在 `VideoAnalyzer` 内收口管理。
- Prompt 构造由统一方法生成，不允许分散在多个模块各自拼接。
- 最终输出 JSON 统一由 CLI 汇总写出。

### 10.3 配置统一管理

新增配置项必须统一进入现有配置体系，建议新增在 `frames`、`audio` 或独立 `analysis` 节点下。

候选配置项：

- `analysis.context_window`
- `analysis.memory_update_interval`
- `analysis.scene_change_threshold`
- `analysis.max_image_side`
- `analysis.audio_alignment_window`
- `analysis.enable_chunk_summary`

约束：

- 不允许新增隐藏配置。
- 不允许把关键策略写死在 prompt 文件中。

## 11. 输出闭环设计

### 11.1 中间产物闭环

系统应明确保留以下中间产物或逻辑对象：

- 关键帧列表
- 帧级结构化分析
- 块级摘要
- 全局剧情记忆
- 音频转写
- 音频摘要

### 11.2 最终结果闭环

最终输出应能够回答：

- 识别了哪些关键画面
- 哪些人物在何时出现
- 关键剧情如何推进
- 音频对白与环境音如何辅助判断
- 最终视频总结如何由中间结果推导而来
- 哪些场景、剧情节点、角色关系与关键对白可以直接作为后续剧本或视频生成的输入素材

建议最终输出增加“剧本辅助素材层”，用于服务后续内容生成：

- `scene_cards`
  - 面向场景拆解
  - 包含场景时间范围、地点/环境、出场人物、关键动作、关键道具、对白摘要
- `story_beats`
  - 面向剧情推进
  - 包含剧情节点顺序、冲突变化、转场点、高潮点、收束点
- `character_timeline`
  - 面向人物线
  - 包含角色首次出现、主要行为、关系变化、称谓变化、离场时点
- `script_guidance`
  - 面向后续剧本/文生视频使用
  - 包含镜头建议、场景描述摘要、对白候选、情绪与节奏建议

### 11.3 失败闭环

当某一环节失败时，系统必须可降级而非整体崩溃：

- 音频失败 -> 继续视觉分析
- 个别帧失败 -> 保留错误记录并继续后续分析
- 块级摘要失败 -> 保留原始帧结果，最终总结继续执行

## 12. 兼容性评估

### 12.1 对 CLI 的兼容性

兼容性结论：

- 兼容。

原因：

- 可保持当前 `video-analyzer video.mp4` 命令不变。
- 新策略通过配置项启用或调整，不破坏现有参数接口。

### 12.2 对 UI 的兼容性

兼容性结论：

- 基本兼容。

原因：

- UI 当前只负责收集参数与展示结果。
- 只要 CLI 参数与输出主结构不被破坏，UI 基本无需大改。

注意：

- 若新增配置项需要在 UI 中可视化配置，则属于增量适配，不属于破坏性改动。

### 12.3 对输出 JSON 的兼容性

兼容性结论：

- 应采用向下兼容扩展。

要求：

- 现有字段保留。
- 新增字段采用可选扩展，如：
  - `frame_analysis[].structured`
  - `chunk_summaries`
  - `story_memory`
  - `audio_summary`

### 12.4 对现有 prompt 的兼容性

兼容性结论：

- 存在一定变更影响。

原因：

- 当前 prompt 假设输入是全历史文本。
- 新方案会改成“局部窗口 + 全局摘要 + 音频摘要”。

结论：

- Prompt 改造属于中等影响改动，需多人评审。

## 13. 破坏性评估

### 13.1 低破坏项

- 增加图片缩放配置
- 增加音频摘要
- 增加块级摘要
- 增加新输出字段

以上属于低破坏扩展。

### 13.2 中破坏项

- 修改逐帧 prompt 构造方式
- 将全历史拼接改为局部窗口 + 全局记忆
- 重构最终总结输入结构

以上会影响输出风格与结果稳定性，属于中破坏改动，需要灰度验证。

### 13.3 高风险项

- 若引入新的人脸识别、目标跟踪、外部角色识别服务
- 若改造为多路并行分析且失去统一状态收口

以上不属于当前需求文档推荐的一阶段范围。

## 14. 分阶段实施建议

### 14.1 第一阶段

目标：

- 先解决性能失控问题，并建立统一状态中心。

范围：

- 有界上下文窗口
- 结构化帧分析
- 音频摘要
- 音频片段对齐
- 图片缩放

### 14.2 第二阶段

目标：

- 提升剧情连贯性与人物一致性。

范围：

- 全局剧情记忆
- 人物注册表
- 场景切换分块
- 块级摘要

### 14.3 第三阶段

目标：

- 做结果质量校验与回归评估。

范围：

- 一致性校验
- 输出质量比对
- 不同模型与配置下的性能评估

## 15. 验收标准

### 15.1 性能验收

- 长视频逐帧平均耗时应趋于稳定，不再随总帧数显著线性增长。
- 长视频分析时不应频繁出现 prompt 截断。

### 15.2 内容验收

- 同一人物在多个时间段中的称谓、服饰、身份描述更稳定。
- 场景切换后的描述不应被旧场景信息污染。
- 音频中的关键对白与画面事件能够形成辅助印证。
- 输出结果能够稳定沉淀出场景卡、剧情节点、人物时间线等剧本辅助素材。
- 输出结果能够为后续视频生成/剧本编写提供足够的结构化依据，而不是仅有一段自由文本总结。

### 15.3 架构验收

- 分析逻辑仍由统一主入口驱动。
- 上下文状态只有统一一处维护。
- 配置、输出、日志路径清晰，无多路重复实现。

## 16. 文档自审结论

### 16.1 是否符合用户功能需求

符合。

原因：

- 已覆盖“图片识别加速”。
- 已覆盖“输出连贯性、人物一致性、剧场需求”。
- 已补充“音频处理与图片识别后的处理规则关联”。
- 已补充“统一入口、统一收口、闭环、兼容性、破坏性评估”。

### 16.2 是否符合统一入口、统一收口原则

符合。

原因：

- 保持 `cli.py` 为唯一业务入口。
- 保持 `VideoAnalyzer` 为分析状态统一收口。
- 配置仍走统一配置体系。
- 输出仍统一由 CLI 汇总写出。

### 16.3 是否形成闭环

符合。

闭环路径如下：

- 视频输入
- 音频抽取/转写
- 关键帧抽取
- 帧级分析
- 音视频联动
- 块级摘要
- 全局记忆沉淀
- 最终总结
- JSON 输出
- 降级失败处理

### 16.4 是否存在信息缺口

存在但可控。

当前尚未在代码中发现以下现成能力：

- 强身份级人物识别
- 人脸 ReID
- 目标级持续跟踪

因此本文档将“人物一致性”定义为基于结构化视觉描述与音频称谓约束的增强一致性，而非强生物特征识别一致性。

### 16.5 是否建议进入实施

建议进入“详细技术设计”阶段，但必须多人评审后实施。

重点评审项：

- 上下文窗口大小
- 音频对齐窗口大小
- 结构化输出字段定义
- 块级摘要更新时机
- 输出 JSON 扩展字段命名
